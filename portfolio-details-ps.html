Overview

Five versions of deliberate over-engineering

Project Storage, a peer-to-peer storage marketplace, is an "Uber for Storage" connecting people with unused space to those who need it. The product concept is simple. But what makes this worth reading is the engineering beneath it: Version 5 represents a full architectural refactor from a 'Fat Model/View' monolith to a Service-Layer Architecture with SOLID compliance, a pluggable allocation engine, independent finite state machines, and a production deployment following Twelve-Factor App methodology.

The application has three user roles:

  • Renter (lists space),
  • Lessee (books space), and
  • Admin (verifies listings).
Each has distinct dashboards, permissions, and workflows.

Architecture

From Fat Models to a Service Layer

Versions 1–4 placed business logic directly in Views and Models. This produced untestable Views (every test required a full HTTP request), bloated Models with non-database logic, and zero reusability across contexts. Version 5 introduced a strict three-layer separation.

Views views.py
HTTP protocol handling only. Receives parameters, checks permissions, renders templates. Delegates all logic to Services. No business rules here.
Services services.py
Pure Python classes containing all business logic. Framework-agnostic. Reusable in HTTP views, CLI commands, background jobs, and unit tests without HTTP fixtures.
BookingService BookingStateService BookingFormService SpaceService SpaceLayoutService InstallationRequestService
Models models.py
Strictly define the data schema and relationships. No calculation logic, no state transition logic. State is represented via models.TextChoices enums.
Allocation Strategy

Best-Fit with Compaction

When a Lessee books space, the system must select which Rack to assign them to. The goal is to prevent storage fragmentation by leaving large, contiguous blocks available for future bookings rather than scattering small gaps across every rack. This logic lives entirely in BookingFormService.get_rack() and can be swapped without touching the booking workflow (Strategy Pattern).

1
Quantify potential. Query all racks in the selected Space, annotating each with its current total available shelf count using Django ORM's Subquery and OuterRef.
2
Filter by need. Exclude any rack whose total available shelf count is less than the number of contiguous shelves the booking requires.
3
Best-fit selection. Sort remaining racks in ascending order by available shelf count — tightest fit first.
4
Compaction. Select .order_by('available_shelves').first() — the rack with the smallest excess capacity. Subsequent logic verifies and marks the specific consecutive shelf block within it.
Key Engineering Contributions

Where the interesting decisions were made

01
Service-Layer Architectural Refactor
Transitioned from monolithic Fat Model/View to a three-layer SOA. Business logic extracted into dedicated service modules, enforcing Separation of Concerns. Services are framework-agnostic: the same BookingService can be called from an HTTP view, a management command, or a unit test. No HTTP request required to validate an allocation.
02
Two Independent State Machines (FSMs)
BookingStateService manages BOOKED → ACTIVE → PAST, triggering occupy_­space() and release_space() side effects on transition. InstallationRequestStateService manages PENDING → APPROVED → COMPLETED, where COMPLETED is the only valid trigger for the Builder Pattern conversion to a live Space. Invalid transitions are blocked by custom exceptions.
03
Builder Pattern — Atomic Space Creation
InstallationRequestService.convert_­to_space() acts as Director/Builder: takes a completed InstallationRequest and atomically constructs the full Location → Space → Rack → Shelf hierarchy before deleting the temporary request. This prevents phantom listings: a Space only exists once it's been physically verified and the full object graph is complete.
04
Pluggable Allocation — Strategy Pattern
The allocation algorithm is decoupled from the booking workflow via the Strategy Pattern. Switching from Best-Fit to First-Fit or Load-Balancing is a change to one method in BookingFormService, not a refactor of the booking view. The current strategy uses ORM-level Subquery/OuterRef annotations for efficiency — no Python-level sorting of large querysets.
05
Composite Pattern — Space Hierarchy
Storage is modelled as a Space → Rack → Shelf composite hierarchy in listings/­models.py. This enables aggregate queries (space.total_shelves sums across all racks recursively) and powers the color-coded grid visualization in SpaceLayout­Service. The visualization logic is extracted into its own service to comply with SRP.
06
Domain Exception Hierarchy
Custom exception hierarchy replaces generic Django exceptions: InstallationRequest­Error (base) → InstallationRequest­ConversionError (specific); SpaceState­Exception; BookingStatusException. Application code depends on abstract error types, not raw transition logic strings (Dependency Inversion applied to error handling).
SOLID Compliance

Each principle with a file reference

S
Single Responsibility
Booking model holds state only. BookingService holds calculation logic. BookingStateService holds transition logic. Pricing can change in BookingService without touching models.py.
SpaceListView delegates filter logic entirely to SpaceService.get_available_spaces(). The View's only responsibility is HTTP protocol handling.
O
Open/Closed
New locations and pricing rules are added to listings/constants.py without modifying SpaceService internals. Django CBVs are extended via get_queryset() and get_context_data() hooks. Django's source is never touched.
L
Liskov Substitution
CustomUser inherits AbstractUser and remains substitutable everywhere Django expects a user object: request.user, ForeignKey, permission checks, while adding domain-specific fields (role, nric_fin).
I
Interface Segregation
Views compose behavior from small Mixins: SpaceCreateView(LoginRequiredMixin, UserPassesTestMixin, CreateView). No massive "SuperView" base class. Each Mixin adds exactly one capability, and classes import only what they explicitly need.
D
Dependency Inversion
Business logic depends on abstractions, not concretions. Booking.status uses models.TextChoices. Callers depend on Booking.Status.ACTIVE, not the string 'ACTIVE'. Error handling depends on the abstract exception hierarchy, not raw status checks. Allocation strategy objects can be injected into services.
Deployment & DevOps

Twelve-Factor App on Render

IaC
Infrastructure as Code
Deployment logic is version-controlled in build.sh, automating dependency installation, static file collection, and database migrations on every deploy. No manual SSH steps.
ENV
Environment Security
SECRET_KEY, DEBUG, and database credentials are decoupled from the codebase via os.environ. dj-database-url auto-switches between SQLite (local) and PostgreSQL (production).
SRV
Production Serving
A Procfile declares the entry point, instructing Render to serve via Gunicorn. Static files served by WhiteNoise. A custom scripts/create_superuser.py auto-provisions admin access during build. No interactive console required.
