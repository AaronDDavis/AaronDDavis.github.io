Overview

Enterprise architecture, without the enterprise infrastructure

The BTO Management System simulates Singapore's Build-To-Order public housing process end-to-end (applicant eligibility checks, officer registration workflows, manager approvals, and enquiry management) through a command-line interface. The engineering constraint that makes it interesting: no database engine or json, strict ECB layering enforced at package level, and a requirement to single-handedly produce a system maintainable by a large team. The custom persistence layer solves a real systems problem (forward-reference resolution during CSV loading) using a two-pass hydration model. Three independent finite state machines govern runtime correctness.

Key Engineering Contributions

Building the database when one isn't allowed

01
Two-Pass In-Memory Relational Graph
CSV files store only IDs, not object references. Loading is split into two phases: Phase 1 hydrates all entities with primitive fields only, constructing O(1) ID-to-object maps. Phase 2 re-parses the same files to resolve cross-entity references and build a bidirectional in-memory object graph, solving forward-reference issues without any database engine.
02
ECB Architecture with Enforced Layer Isolation
Strict Entity-Control-Boundary separation is enforced structurally, not just by convention. Entity classes have zero imports from userctrl, databasemgr, or display packages. Business logic lives exclusively in Control. I/O exclusively in Boundary. Violations are detectable at compile time.
03
Role-Based State Machines (3 FSMs)
Three independent FSMs govern runtime state: the Application FSM with per-subclass transition rules; the Applicant FSM with boolean guards (isWithdrawing, etc); and the Officer FSM tracking registered/joined/prohibited project sets with date-overlap eligibility checks.
04
Role-Scoped Repository — Strategy + Repository Patterns
Database<T> is a generic in-memory repository. Access is mediated by role-specific manager classes (ProjectDatabase­Mgr, ApplicationDatabaseMgr) whose getData() methods apply role filters automatically. Managers see all, officers see non-prohibited, applicants see eligibility-filtered results. The same query interface, different results.
05
Consistent UI Rendering — Template Method
Abstract ItemDisplayer<T> defines a fixed rendering algorithm: dis­play(List<T>) calls abstract display(T) per item. Subclasses override only the entity-specific step. This enforces a consistent UI contract across all entity types while eliminating duplicated formatting logic.
06
Facade + Factory for System Orchestration
BTOManagementSystem acts as a Facade, coordinating the two-phase load/save sequence across 6 entity types without exposing that complexity to callers. Object creation uses a Factory: Application­Mgr.create() switches on an APPLICATION_TYPE enum, decoupling instantiation from client code and keeping constructors private to their domains.
